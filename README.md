# CodeMentor AI

Гибридный AI-ревьюер Python-кода: детерминированный статический анализ + LLM-слой
для логических и архитектурных замечаний. Capstone-проект, Track 3 (AI Application/API).

## Проблема и пользователь

Junior-разработчики и студенты часто не получают быстрый, структурированный фидбэк
по своему коду до собеседования, code review в команде или публикации портфолио.
CodeMentor AI — это бесплатный «второй взгляд senior-разработчика»: вставляешь код,
получаешь список конкретных проблем с объяснением и предложенным исправлением.

## Почему гибрид, а не просто вызов LLM

Чистый вызов LLM («просто спросить модель про код») — ненадёжен: он может
галлюцинировать несуществующие проблемы, пропускать очевидные баги или давать
неверные номера строк. Поэтому решение состоит из двух слоёв:

1. **Статический анализатор** (`reviewer/static_analyzer.py`) — собственный обход
   AST на чистом stdlib Python, без сторонних библиотек. Ловит конкретный список
   паттернов: SQL-инъекции (включая случай через промежуточную переменную),
   `eval`/`exec`, хардкод секретов, мутабельные дефолтные аргументы, голые
   `except:`, сравнение с `None` через `==`, затенение builtin-имён, высокую
   цикломатическую сложность, длинные строки. 100% детерминированный, ноль
   галлюцинаций, ноль стоимости, работает офлайн.
2. **LLM-слой** (`reviewer/llm_reviewer.py`) — получает код и находки слоя 1 как
   контекст, ищет то, что статика принципиально не видит: логические ошибки,
   неудачные названия, архитектурные замечания. Использует бесплатный API Groq
   (Llama 3.3 70B).

`reviewer/aggregator.py` объединяет находки обоих слоёв и считает итоговый score
(100 минус штраф за каждую находку, штраф зависит от severity — прозрачная,
проверяемая на бумажке формула, не «магическое число»).

## Baseline для сравнения

Намеренно простое решение: один вызов LLM с минимальным неструктурированным
промптом («Посмотри на этот код и скажи, есть ли в нём проблемы»), без
статического контекста, без JSON-схемы. Реализован в `llm_reviewer.review_baseline()`
и используется только в `tests/evaluate.py --mode live` для честного сравнения —
именно так выглядело бы наивное «обёрточное» решение поверх API.

## Тестовый набор и метрики

`tests/fixtures/` — 13 Python-файлов с заранее известными, специально заложенными
проблемами (`tests/ground_truth.json`), включая:
- файл с полностью чистым кодом (тест на false positives),
- файл с чисто логической ошибкой, которую *по архитектуре* не может поймать
  статический слой — это намеренно, чтобы границы каждого слоя были видны явно,
- edge cases: пустой файл, синтаксически некорректный код.

`tests/evaluate.py` прогоняет оба решения (baseline и main) на всём наборе и
считает per-category precision/recall/F1 относительно ground truth, а не просто
accuracy (которая была бы бессмысленна на несбалансированном наборе категорий).

```bash
python -m tests.evaluate --mode mock   # быстрая проверка пайплайна без сети
python -m tests.evaluate --mode live   # реальные метрики с GROQ_API_KEY (для отчёта)
```

Важно: в `--mode mock` LLM-слой не вызывает реальную модель (заглушка нужна для
разработки без сети и без расхода лимита API), поэтому в mock-режиме baseline
не оценивается вообще — оценивать заглушку как baseline было бы нечестным
сравнением. Честные числа для Final Report получаются только в `--mode live`.

## Ограничения (честно)

- Поддерживается только Python.
- Статический слой — это набор явных правил, а не полноценный linter; он не
  претендует на покрытие всех возможных проблем.
- LLM-слой может давать ложноположительные срабатывания и путать номера строк.
- Taint-tracking для SQL-инъекций облегчённый, работает в пределах одной функции.
- Метрики посчитаны на собственном контролируемом наборе из 13 файлов — это
  оценка качества системы разработчиками, а не строгая гарантия на любом
  стороннем коде.
- Бесплатный тир Groq имеет rate limits — при проверке многих файлов подряд
  возможны задержки или временный fallback на static-only.

## Как запустить

### Локально

```bash
git clone https://github.com/YOUR_USERNAME/codementor-ai.git
cd codementor-ai
conda create -n codementor python=3.11 -y && conda activate codementor
pip install -r requirements.txt
cp .env.example .env   # впишите GROQ_API_KEY (см. ниже, как получить)
python app.py
```

Откроется на `http://127.0.0.1:7860`.

### Через Docker

```bash
docker compose up --build
```

### В Google Colab

См. `notebooks/demo_colab.ipynb` — запускается с чистого листа, без локальной
установки чего-либо.

### Как получить бесплатный API-ключ

1. https://console.groq.com → вход по email или Google-аккаунту, без карты.
2. Создать API key в разделе API Keys.
3. Вставить в `.env` как `GROQ_API_KEY=...`.

## Структура репозитория

```
codementor-ai/
├── app.py                  # Gradio веб-интерфейс
├── reviewer/
│   ├── static_analyzer.py  # Слой 1: детерминированный AST-анализ
│   ├── llm_reviewer.py     # Слой 2: LLM-ревью + baseline
│   └── aggregator.py       # Объединение слоёв, расчёт score
├── tests/
│   ├── fixtures/            # 13 тестовых файлов с известными проблемами
│   ├── ground_truth.json    # Разметка для расчёта метрик
│   ├── evaluate.py           # Расчёт precision/recall/F1
│   └── results/metrics.json # Сохранённые результаты последнего прогона
├── notebooks/demo_colab.ipynb
├── docs/                    # Action Plan, WBS, Final Report
├── Dockerfile, docker-compose.yml
└── requirements.txt
```

## Стек

Python 3.11, Gradio (UI), Groq API / Llama 3.3 70B (бесплатный тир), стандартная
библиотека Python (`ast`) для статического анализа — без тяжёлых ML-зависимостей,
весь проект воспроизводим на бесплатных мощностях.
